# План переноса с Python/FastAPI на Bitrix24-версию

## Этап 1: документация и анализ

### Цель

Формализовать текущую предметную область и зафиксировать целевую архитектуру.

### Результат

Комплект документов по бизнес-логике, модели данных, API и Bitrix24-интеграции.

### Критерии готовности

- есть согласованный README новой версии
- описаны сущности, тарифные блоки и ключевые endpoint'ы
- все пробелы помечены `TODO` или `NEEDS_CONFIRMATION`

### Риски

- неполный перенос знаний из legacy-кода
- неуточненные формулы расчета

## Этап 2: React + NestJS каркас

### Цель

Создать базовый monorepo и инфраструктурный скелет проекта.

### Результат

- React frontend
- NestJS backend
- общие пакеты типов
- базовые dev/test команды

### Критерии готовности

- проект собирается локально
- frontend и backend поднимаются одной командой или согласованным набором команд

### Риски

- неудачный выбор monorepo tooling
- избыточная сложность dev setup

## Этап 3: Bitrix24 placement

### Цель

Встроить приложение во вкладку сделки CRM.

### Результат

- реализован install flow
- реализован placement `CRM_DEAL_DETAIL_TAB`
- приложение открывается из карточки сделки

### Критерии готовности

- приложение устанавливается в тестовый портал
- при открытии вкладки корректно определяется `dealId`

### Риски

- ошибки install payload
- проблемы iframe/HTTPS/CORS

## Этап 4: PostgreSQL + Prisma модели

### Цель

Создать целевую БД и Prisma schema.

### Результат

- Prisma schema
- миграции
- базовые seed-данные справочников

### Критерии готовности

- БД разворачивается с нуля
- ключевые сущности доступны через Prisma

### Риски

- неправильная декомпозиция тарифных сущностей
- недостающие индексы

## Этап 5: базовый калькулятор

### Цель

Собрать минимальный end-to-end сценарий расчета.

### Результат

- форма расчета во вкладке сделки
- расчет базового сценария
- сохранение расчета

### Критерии готовности

- пользователь может выполнить расчет без ручного вызова API
- результат сохраняется в БД

### Риски

- незавершенные входные формы
- несогласованность DTO frontend/backend

## Этап 6: перенос тарифной логики

### Цель

Перенести логику подбора и применения тарифов из legacy-системы.

### Результат

- реализованы все тарифные блоки MVP
- расчет учитывает этапы и дополнительные блоки

### Критерии готовности

- результаты по контрольным кейсам сопоставимы с legacy-версией

### Риски

- скрытая логика в `services.py`
- неописанные исключения бизнес-правил

## Этап 7: импорт/экспорт XLSX

### Цель

Перенести пакетную загрузку и выгрузку тарифов.

### Результат

## Этап 8: PDF

### Цель

Реализовать экспорт расчета в PDF.

### Результат

- backend формирует PDF
- PDF можно открыть из расчета и/или сохранить ссылку в сделке

### Критерии готовности

### Риски

- требования к фирменному шаблону

Подготовить production-развертывание.
